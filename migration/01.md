# Open-Meteo API Fetch Layer Implementation

**Date:** 2026-03-03  
**Scope:** Build standalone `server/open-meteo/` module — no integration into API handler or frontend  
**Goal:** Fetch wind + wave + sunrise/sunset from Open-Meteo, return same shape as Windguru layer

---

## Background

Windguru's Terms prohibit automated access. The forecast models themselves (GFS, ICON, GDPS, IFS-HRES) are public domain data from government weather agencies. Open-Meteo serves the same model data via a free, documented REST API with explicit open-source-friendly terms (CC-BY 4.0).

This plan adds Open-Meteo as a **second data source** alongside Windguru. The Open-Meteo layer will be standalone and unused until a future integration phase.

---

## API Reference

### Wind Forecast Endpoint

```
GET https://api.open-meteo.com/v1/forecast
  ?latitude={lat}
  &longitude={lon}
  &hourly=wind_speed_10m,wind_direction_10m,wind_gusts_10m
  &daily=sunrise,sunset
  &wind_speed_unit=kn
  &timezone={tz}
  &forecast_days=7
  &models={model}
```

Response:

```json
{
  "hourly": {
    "time": ["2026-03-03T00:00", "2026-03-03T01:00", ...],
    "wind_speed_10m": [12.5, 14.2, ...],
    "wind_direction_10m": [270, 285, ...],
    "wind_gusts_10m": [18.1, 20.3, ...]
  },
  "daily": {
    "time": ["2026-03-03"],
    "sunrise": ["2026-03-03T06:15"],
    "sunset": ["2026-03-03T18:30"]
  }
}
```

### Marine Forecast Endpoint

```
GET https://marine-api.open-meteo.com/v1/marine
  ?latitude={lat}
  &longitude={lon}
  &hourly=wave_height
  &timezone={tz}
  &forecast_days=7
```

Response:

```json
{
  "hourly": {
    "time": ["2026-03-03T00:00", "2026-03-03T01:00", ...],
    "wave_height": [0.8, 0.9, ...]
  }
}
```

### Error Response

```json
{
  "error": true,
  "reason": "Cannot initialize WeatherVariable from invalid String value..."
}
```

### 2. `server/open-meteo/fetch.ts`

Low-level HTTP client for Open-Meteo. Parallel to `server/scraper/fetch.ts`.

```typescript
import { ApiError } from "../scraper/fetch.js";

const FORECAST_BASE_URL = "https://api.open-meteo.com/v1/forecast";
const MARINE_BASE_URL = "https://marine-api.open-meteo.com/v1/marine";
const REQUEST_TIMEOUT_MS = 10_000;

/**
 * Generic GET request for Open-Meteo APIs.
 * Throws ApiError on HTTP failure or JSON parse failure.
 */
export async function makeRequest<T>(url: string): Promise<T> {
  let response: Response;

  try {
    response = await fetch(url, {
      signal: AbortSignal.timeout(REQUEST_TIMEOUT_MS),
    });
  } catch (error) {
    const isTimeout =
      error instanceof DOMException &&
      (error.name === "TimeoutError" || error.name === "AbortError");

    if (isTimeout) {
      throw new ApiError(`Request timed out after ${REQUEST_TIMEOUT_MS}ms`, { url });
    }
    throw new ApiError(`Network error: ${error instanceof Error ? error.message : String(error)}`, {
      url,
    });
  }

  if (!response.ok) {
    throw new ApiError(`API request failed: ${response.status} ${response.statusText}`, {
      status: response.status,
      response: await response.text().catch(() => "<response body unavailable>"),
      url,
    });
  }

  let json: any;
  try {
    json = await response.json();
  } catch {
    throw new ApiError(`Invalid JSON response (status ${response.status})`, {
      status: response.status,
      url,
    });
  }

  // Open-Meteo returns { error: true, reason: "..." } on validation errors
  if (json && typeof json === "object" && json.error === true) {
    throw new ApiError(`Open-Meteo error: ${json.reason || "Unknown error"}`, {
      status: response.status,
      response: JSON.stringify(json),
      url,
    });
  }

  return json as T;
}

/**
 * Fetch wind forecast + sunrise/sunset from Open-Meteo Forecast API.
 */
export async function fetchForecast(
  lat: number,
  lon: number,
  model: string,
  tz: string,
): Promise<OpenMeteoForecastResponse> {
  const params = new URLSearchParams({
    latitude: lat.toString(),
    longitude: lon.toString(),
    hourly: "wind_speed_10m,wind_direction_10m,wind_gusts_10m",
    daily: "sunrise,sunset",
    wind_speed_unit: "kn",
    timezone: tz,
    forecast_days: "7",
    models: model,
  });

  return makeRequest<OpenMeteoForecastResponse>(`${FORECAST_BASE_URL}?${params}`);
}

/**
 * Fetch wave height from Open-Meteo Marine API.
 */
export async function fetchMarine(
  lat: number,
  lon: number,
  tz: string,
): Promise<OpenMeteoMarineResponse> {
  const params = new URLSearchParams({
    latitude: lat.toString(),
    longitude: lon.toString(),
    hourly: "wave_height",
    timezone: tz,
    forecast_days: "7",
  });

  return makeRequest<OpenMeteoMarineResponse>(`${MARINE_BASE_URL}?${params}`);
}
```

**Design notes:**

- Reuses `ApiError` from `server/scraper/fetch.ts` — it's a generic HTTP error type, no need to duplicate
- Open-Meteo returns `{ error: true, reason: "..." }` even on 200 responses for validation errors — check for this after JSON parse
- No auth headers needed (free tier, no API key)
- Wind speed unit set to knots at API level (`wind_speed_unit=kn`) — no conversion needed

### 3. `server/open-meteo/forecast.ts`

Main orchestration layer. Public API matching `fetchWindData` from `api-scraper.ts`.

```typescript
import type { WindConditionRaw } from "../types/wind-conditions.js";
import type { OpenMeteoForecastResponse, OpenMeteoMarineResponse } from "./types.js";
import { fetchForecast, fetchMarine } from "./fetch.js";
import { tryCatch } from "../utils/try-catch.js";

/**
 * Parse ISO 8601 timestamp to "HH:MM" string.
 * Example: "2026-03-03T06:15" → "06:15"
 */
function extractTime(isoTimestamp: string): string {
  const match = isoTimestamp.match(/T(\d{2}:\d{2})/);
  if (!match) {
    throw new Error(`Invalid ISO timestamp format: ${isoTimestamp}`);
  }
  return match[1];
}

/**
 * Convert Open-Meteo forecast response to WindConditionRaw[].
 */
function extractWindData(forecast: OpenMeteoForecastResponse): WindConditionRaw[] {
  const { time, wind_speed_10m, wind_direction_10m, wind_gusts_10m } = forecast.hourly;

  if (!time || !wind_speed_10m) {
    throw new Error("Missing required forecast data (time or wind_speed_10m)");
  }

  const windData: WindConditionRaw[] = [];

  for (let i = 0; i < time.length; i++) {
    windData.push({
      date: new Date(time[i]),
      windSpeed: wind_speed_10m[i] ?? null,
      windDirection: wind_direction_10m?.[i] ?? null,
      windGusts: wind_gusts_10m?.[i] ?? null,
      waveHeight: null,
    });
  }

  return windData;
}

/**
 * Merge wave data into wind data in-place by matching timestamps.
 */
function mergeWaveData(
  windData: WindConditionRaw[],
  marineResponse: OpenMeteoMarineResponse,
): void {
  const { time, wave_height } = marineResponse.hourly;

  if (!time || !wave_height) return;

  // Build timestamp → waveHeight map
  const waveMap = new Map<number, number>();
  for (let i = 0; i < time.length; i++) {
    const timestamp = new Date(time[i]).getTime();
    const height = wave_height[i];
    if (height != null) {
      waveMap.set(timestamp, height);
    }
  }

  // Match by timestamp
  for (const condition of windData) {
    const timestamp = condition.date.getTime();
    const height = waveMap.get(timestamp);
    if (height !== undefined) {
      condition.waveHeight = height;
    }
  }
}

/**
 * Fetch wind + wave + sunrise/sunset from Open-Meteo.
 *
 * @param lat Latitude (WGS84)
 * @param lon Longitude (WGS84)
 * @param model Open-Meteo model slug (e.g. "gfs_global", "icon_global")
 * @param tz Timezone (e.g. "Asia/Jerusalem")
 * @returns Wind data + sunrise/sunset in "HH:MM" format
 */
export async function fetchOpenMeteoData(
  lat: number,
  lon: number,
  model: string,
  tz: string,
): Promise<{
  windData: WindConditionRaw[];
  sunrise: string;
  sunset: string;
}> {
  // Fetch wind and wave models in parallel — wave failure is non-fatal
  const [windResult, waveResult] = await Promise.all([
    tryCatch(fetchForecast(lat, lon, model, tz)),
    tryCatch(fetchMarine(lat, lon, tz)),
  ]);

  if (windResult.error) {
    throw windResult.error;
  }

  const forecast = windResult.data;

  // Extract sunrise/sunset from daily data
  if (!forecast.daily?.sunrise?.[0] || !forecast.daily?.sunset?.[0]) {
    throw new Error("Missing sunrise/sunset data in forecast response");
  }

  const sunrise = extractTime(forecast.daily.sunrise[0]);
  const sunset = extractTime(forecast.daily.sunset[0]);

  // Convert forecast to WindConditionRaw[]
  const windData = extractWindData(forecast);

  // Merge wave data if available
  if (waveResult.error) {
    console.error(
      `Wave data fetch failed for ${lat},${lon} (non-fatal): ${waveResult.error.message}`,
    );
  } else {
    mergeWaveData(windData, waveResult.data);
  }

  return { windData, sunrise, sunset };
}
```

**Design notes:**

- Mirrors `fetchWindData` signature from `api-scraper.ts` but takes `lat/lon` + `model` instead of `spotId` + `modelId`
- Wave failure is non-fatal (same pattern as Windguru layer)
- Sunrise/sunset are parsed from ISO timestamps to `"HH:MM"` strings to match existing `filterEvents` contract
- No model fallback — if requested model fails, propagate error

## Design Decisions

### 1. Sunrise/Sunset Format

**Issue:** Windguru returns `"HH:MM"` strings. Open-Meteo returns full ISO timestamps like `"2026-03-03T06:15"`.

**Solution:** Parse ISO string with regex `/T(\d{2}:\d{2})/` and extract `"HH:MM"`. This matches the existing contract expected by `filterEvents`.

### 2. Wave Data Merge Strategy

**Issue:** Wind and wave data come from separate endpoints. How to merge?

**Solution:** Same strategy as Windguru layer:

1. Build `Map<timestamp, waveHeight>` from marine response
2. Iterate wind data, match by timestamp (`date.getTime()`)
3. Mutate `waveHeight` field in-place

Both APIs return ISO strings in the same timezone, so timestamp matching is reliable.

### 3. Wave Data for Inland Spots

**Issue:** Sea of Galilee (Kinneret) and potentially Eilat (Red Sea) may not have marine API coverage.

**Solution:** Wave failure is non-fatal (same as Windguru). If marine API returns an error or no data, log it and continue. The `waveHeight` field stays `null`.

### 4. Error Handling

**Issue:** Should we create a new error class or reuse `ApiError`?

**Solution:** Reuse `ApiError` from `server/scraper/fetch.ts`. It's already a generic HTTP error type with `status`, `response`, `url` fields. No need to duplicate. Import it directly.

### 5. Model Fallback

**Issue:** Windguru layer falls back to GFS (model 3) if the requested model fails. Should Open-Meteo do the same?

**Solution:** No fallback. If the requested model fails, propagate the error. The caller can decide fallback strategy during integration. Keeps the layer simple and predictable.

### 6. Forecast Days

**Issue:** Open-Meteo supports up to 16 days. Windguru layer fetches 7 days.

**Solution:** Use `forecast_days=7` to match current behavior. This can be made configurable later.

## Key Simplifications vs Windguru Layer

1. **No 2-step flow** — Windguru requires spot info → extract `rundef` → fetch forecast. Open-Meteo is a single request per endpoint.
2. **No timestamp math** — Windguru uses `initstamp + hours[]` offset scheme. Open-Meteo returns ISO strings directly.
3. **No unit conversion** — Wind speed unit set to knots at API level (`wind_speed_unit=kn`). Windguru returns m/s and requires conversion.
4. **No `spotId` mapping** — Takes `lat/lon` directly. The integration layer will resolve location → coordinates.

## Open Questions

### 1. Error Class

Reuse `ApiError` from `server/scraper/fetch.ts` or create `OpenMeteoApiError`?

**Recommendation:** Reuse. It's a generic HTTP error type, no coupling.

### 2. Marine API Coverage

What happens when marine API has no data for inland spots (Sea of Galilee)?

**Assumption:** API will return an error or empty data. The non-fatal pattern handles this — we log and continue with `waveHeight: null`.

### 3. Sunrise/Sunset on Day 2+

Open-Meteo's `daily` array has 7 entries (one per day). Current code only uses `daily.sunrise[0]` / `daily.sunset[0]` (first day).

**Issue:** Windguru returns a single sunrise/sunset per forecast (presumably for the forecast's reference day). Do we need per-day sunrise/sunset?

**Assumption:** The current Windguru layer's `filterEvents` uses a single sunrise/sunset for all days in the forecast. This is incorrect for multi-day forecasts (sunrise changes by ~1 min/day), but matches existing behavior. We'll replicate this for consistency. A future improvement could pass per-day sunrise/sunset to `filterEvents`.
