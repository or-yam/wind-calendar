# Open-Meteo Integration Phase

**Date:** 2026-03-03  
**Depends on:** Phase 1 (Open-Meteo fetch layer — `migration/01.md`)  
**Scope:** Integrate Open-Meteo as a selectable provider alongside Windguru  
**Goal:** Users can choose Open-Meteo or Windguru via frontend dropdown, with coordinated fallback logic

---

## Background

Phase 1 built a standalone `server/open-meteo/` module that fetches wind + wave + sunrise/sunset from Open-Meteo. This phase integrates it into the application:

1. Add `lat/lon` coordinates to all locations
2. Add provider routing in API handler (Windguru vs Open-Meteo)
3. Update frontend to allow provider selection with grouped model dropdown
4. Add Open-Meteo attribution in UI
5. Implement provider-aware fallback (Open-Meteo → Windguru, not the reverse)

---

## Design Decisions

### 1. Provider Selection: Query Param vs Model Namespace

**Option A:** Add `?provider=windguru|openmeteo` query param, keep existing model IDs (3, 45, 59, 117)

**Option B:** Use model ID namespaces: `om_gfs`, `om_icon`, `wg_gfs_3`, `wg_icon_45`

**Decision:** Option B (model ID namespaces)

**Rationale:**

- Single source of truth — model ID encodes both provider + model
- No dual-param validation (`?model=3&provider=openmeteo` would be invalid)
- Backward compatible — existing URLs with `?model=3` continue to work
- Frontend can naturally group models by provider in dropdown
- Simpler API handler — no provider routing logic, just model → provider inference

**Model ID Mapping:**

| Model         | Windguru ID (legacy) | Open-Meteo ID (new) |
| ------------- | -------------------- | ------------------- |
| GFS 13 km     | `3` (keep)           | `om_gfs`            |
| ICON 13 km    | `45` (keep)          | `om_icon`           |
| GDPS 15 km    | `59` (keep)          | `om_gdps`           |
| IFS-HRES 9 km | `117` (keep)         | `om_ifs`            |

**URL Examples:**

- Legacy (Windguru): `?location=tel-aviv&model=3` ✓
- Open-Meteo: `?location=tel-aviv&model=om_gfs` ✓

### 2. Fallback Strategy

**Issue:** Should we fall back between providers if one fails?

**Decision:** Open-Meteo → Windguru fallback only (not bidirectional)

**Rationale:**

- Open-Meteo has no ToS restrictions, but Windguru prohibits automated access
- If user explicitly selects Windguru (legacy model IDs 3/45/59/117), honor that choice — no fallback to Open-Meteo
- If user selects Open-Meteo and it fails (outage, API error), fall back to Windguru as emergency source
- Log fallback events for monitoring

**Fallback flow:**

```
User selects model=om_gfs
  → Try Open-Meteo (gfs_global)
    → Success: return data
    → Failure: Try Windguru (model 3)
      → Success: return data + warning header
      → Failure: return 502 error

User selects model=3 (Windguru GFS)
  → Try Windguru (model 3)
    → Success: return data
    → Failure: return 502 error (no fallback)
```

**Implementation:**

- Add `X-Data-Source: windguru` or `X-Data-Source: openmeteo` response header
- Add `X-Fallback-Used: true` header when fallback occurs
- Log fallback events to console in dev, structured logs in prod

### 3. Location Coordinates Source

**Issue:** Where do we get `lat/lon` for each location?

**Options:**

1. Manually geocode each location (Google Maps, OpenStreetMap)
2. Fetch from Windguru spot info API (already returns `spot_lat`, `spot_lon`)
3. Hybrid: Use Windguru API if available, fallback to manual

**Decision:** Option 2 (Windguru spot info API)

**Rationale:**

- We already call `fetchSpotInfo(spotId)` in Windguru flow — coordinates are there
- Guarantees Open-Meteo fetches for the exact same point as Windguru
- No manual geocoding errors
- If spot info fails, Open-Meteo provider is unavailable (acceptable degradation)

**Implementation:**

- Add `coordinates?: { lat: number; lon: number }` field to `LocationConfig` type
- Populate coordinates from Windguru spot info response during Windguru fetch
- Cache coordinates for 24h (longer than forecast cache) — coordinates don't change
- If Open-Meteo is selected and coordinates are missing, return 500 with clear error

### 4. Model Availability Per Location

**Issue:** Should we show Open-Meteo models for all locations, or check coverage first?

**Decision:** Show all Open-Meteo models for all locations (no coverage check)

**Rationale:**

- Open-Meteo has global coverage for all 4 models (GFS, ICON, GDPS, IFS)
- Wave data (marine API) may fail for inland spots (Sea of Galilee), but that's already handled as non-fatal
- Simplifies frontend logic — no dynamic filtering needed
- If a model fails for a specific location, fallback handles it gracefully

**Implementation:**

- All locations get both Windguru + Open-Meteo model options
- Existing `models: [3, 45, 59, 117]` field in locations config stays for Windguru-only compatibility (if needed)
- Frontend shows all 8 models (4 Windguru + 4 Open-Meteo) grouped by provider

### 5. `api/calendar.ts`

Add provider routing logic:

```typescript
import { fetchWindData } from "../server/scraper/api-scraper.js";
import { fetchOpenMeteoData } from "../server/open-meteo/forecast.js";
import { getProvider, isOpenMeteoModel, getWindguruFallback, MODELS } from "../shared/models.js";

// ... existing imports

export default async function handler(req: VercelRequest, res: VercelResponse) {
  // ... existing rate limit, query parsing

  const location = resolveLocation(config.location);
  const provider = getProvider(config.model);

  let fetchResult: Awaited<ReturnType<typeof fetchWindData>> | null = null;
  let dataSource: "windguru" | "openmeteo" = provider;
  let fallbackUsed = false;

  if (provider === "openmeteo") {
    // Open-Meteo path with Windguru fallback
    if (!location.coordinates) {
      return res.status(500).json({
        error: "Location coordinates unavailable",
        code: "MISSING_COORDINATES",
        suggestion: "This location does not support Open-Meteo provider",
      });
    }

    const openMeteoModel = MODELS[config.model];
    if (!("openMeteoSlug" in openMeteoModel)) {
      return res.status(500).json({ error: "Invalid Open-Meteo model configuration" });
    }

    const { data: omData, error: omError } = await tryCatch(
      fetchOpenMeteoData(
        location.coordinates.lat,
        location.coordinates.lon,
        openMeteoModel.openMeteoSlug,
        location.tz,
      ),
    );

    if (omError) {
      // Fallback to Windguru
      console.error(
        `Open-Meteo fetch failed for ${config.location}/${config.model}: ${omError.message}`,
      );
      const fallbackModelId = getWindguruFallback(config.model);

      if (fallbackModelId) {
        console.log(`Falling back to Windguru model ${fallbackModelId}`);
        const { data: wgData, error: wgError } = await tryCatch(
          fetchWindData(location.spotId, fallbackModelId),
        );

        if (wgError) {
          const { status, body } = classifyFetchError(wgError, location.spotId, isDev);
          return res.status(status).json(body);
        }

        fetchResult = wgData;
        dataSource = "windguru";
        fallbackUsed = true;
      } else {
        // No fallback available
        return res.status(502).json({
          error: "Open-Meteo request failed",
          code: "OPENMETEO_FAILED",
          suggestion: "Try again later or select a Windguru model",
          ...(isDev && { debug: { message: omError.message } }),
        });
      }
    } else {
      fetchResult = omData;
      dataSource = "openmeteo";
    }
  } else {
    // Windguru path (no fallback)
    const { data: wgData, error: wgError } = await tryCatch(
      fetchWindData(location.spotId, config.model as number),
    );

    if (wgError) {
      const { status, body } = classifyFetchError(wgError, location.spotId, isDev);
      return res.status(status).json(body);
    }

    fetchResult = wgData;
    dataSource = "windguru";
  }

  // ... existing buildCalendar logic

  res.setHeader("Content-Type", "text/calendar; charset=utf-8");
  res.setHeader("X-Data-Source", dataSource);
  if (fallbackUsed) {
    res.setHeader("X-Fallback-Used", "true");
  }
  res.setHeader(
    "Cache-Control",
    "public, max-age=21600, stale-while-revalidate=86400, stale-if-error=604800",
  );
  res.setHeader("Content-Disposition", `inline; filename="wind-forecast-${config.location}.ics"`);
  return res.status(200).send(icsString);
}
```

## Open Questions

### 1. Default Provider

Should new users default to Open-Meteo or Windguru?

**Recommendation:** Open-Meteo (use `om_gfs` as default model in frontend). It has no ToS restrictions and supports the project's long-term sustainability.

### 2. Deprecation Timeline for Windguru

Should we deprecate Windguru models after Open-Meteo proves stable?

**Recommendation:** Keep both indefinitely. Windguru provides redundancy. Add a notice in UI: "We recommend Open-Meteo models for best reliability."

### 3. Coordinate Precision

Windguru spot info returns coordinates with 3 decimal places (e.g., `32.085`). Is this precise enough?

**Assumption:** Yes. 3 decimal places ≈ 111m precision. Weather models have 9-13km resolution, so coordinate precision is not a limiting factor.

### 4. Open-Meteo Rate Limits

Free tier: 10K calls/day, 5K/hour, 600/min. What if we exceed?

**Mitigation:**

- Current usage: ~288 calls/day (well within limits)
- CDN cache: 6h TTL reduces load
- If limits are hit, fallback to Windguru (handled gracefully)
- Monitor usage via logs

This phase integrates Open-Meteo as a selectable provider:

1. **Model ID namespace:** Use `om_gfs`, `om_icon`, `om_gdps`, `om_ifs` for Open-Meteo; keep `3`, `45`, `59`, `117` for Windguru
2. **Provider routing:** API handler infers provider from model ID, routes to `fetchOpenMeteoData` or `fetchWindData`
3. **Fallback:** Open-Meteo → Windguru (not bidirectional), with headers `X-Data-Source` and `X-Fallback-Used`
4. **Coordinates:** Hardcode `lat/lon` in `shared/locations.ts` (fetched once from Windguru spot info API)
5. **Frontend:** Grouped model dropdown (Windguru / Open-Meteo), Open-Meteo marked as recommended
6. **Attribution:** Add Open-Meteo link in footer (CC-BY 4.0 requirement)
7. **Backward compatibility:** Existing URLs with `?model=3` continue to work
